# 举报处理系统 需求文档

## 1. 背景与现状
- 现状：测试版举报系统已经上线并合并代码。用户可以从回复跳转到举报中心提交举报，举报按 `post_id` 合并展示，管理员可以在举报中心执行提示、忽略、删帖、删全、碎饼、封禁饼干等操作。
- 新问题：不同管理员只负责部分板块。现有举报中心需要按被举报回复所属 `forum_id` 做展示分类和管理员鉴权，避免管理员看到或操作自己无权限板块的处理入口。
- 权限来源：数据库 `admins.forums` 字段，字段类型为 JSON，Laravel `Admin` 模型已 cast 为 array。本地 `dao.admins` 示例中该字段形如 `[2932]`、`[1229]`、`[1, 2, 10, 12, ...]`。管理员等级仍来自 `users.admin` / token ability，板块范围只来自 `admins.forums`。

## 2. 目标
- 举报记录只保存 `forum_id`，不冗余保存 forum 名称。
- 举报中心每条举报展示所属板块名称，名称由前端通过 `resources/stores/forums.ts` 根据 `forum_id` 实时解析。
- 管理员 UI 和后端管理员操作都必须按单条举报的 `forum_id` 做权限判断。
- 原“只显示未处理”筛选改为“只显示我的待处理”，即只显示当前管理员有权限板块内的未处理举报。

## 3. 需求分析与约定
- 需求整体可实现，且与当前数据结构一致：`accuses` 表已经有 `forum_id` 字段，提交举报时可以从被举报 `post.forum_id` 写入。
- 前端不能只使用全局 `isForumAdmin` 判断整页管理员状态。必须按每条举报的 `forum_id` 判断当前用户是否有该板块权限，否则拥有任意板块权限的管理员会在其他板块举报上看到处理入口。
- 后端不能依赖前端隐藏按钮来保证安全。所有管理员接口都必须用举报记录自身的 `forum_id` 和当前用户的 `admins.forums` 做鉴权。
- “只显示我的待处理”的口径约定为：`status = pending` 且 `accuses.forum_id in 当前管理员 admins.forums`。普通用户或没有任何板块权限的用户开启该筛选时应返回空列表；前端也可以只在有管理员权限时展示这个筛选。
- “我的待处理”不是“我本人创建的举报”，也不是“分配给我的举报”。当前系统没有举报分配字段。
- 匿名约束不变：举报人和被举报人身份信息对所有角色均不可见，包括管理员。

## 4. 基本使用流程
1. 举报流程
   1. 在回复组件 `resources/vue/Thread/PostItem/PostItem.vue` 的举报按钮跳转至 `/accuse?thread_id=xxx&post_id=xxx&floor=xxx`。
   2. 举报中心根据 query 自动弹出新增举报 modal，自动填写 `thread_id`、`post_id` 和 `floor`。
   3. 一个举报单只对应一个回复。举报记录以 `post_id` 作为唯一标识，`floor` 仅作为展示和跳转辅助信息。
   4. 举报人需填写不少于 5 字的举报理由。
   5. 同一用户对同一个 post 只能举报 1 次。不同用户举报同一个 post 时，合并到同一个举报项，保留每次举报理由。
   6. 后端创建或更新举报项时，从被举报回复读取并写入 `forum_id`，不接受前端传入的 `forum_id` 作为可信来源。
   7. 提交时前端校验：`post_id`、`thread_id` 为有效正整数，`floor` 为有效非负整数，举报理由不少于 5 字。后端校验：主题不存在、回复不存在、回复不属于该主题、楼层不匹配、同一用户重复举报同一 post。
   8. 提交或取消后清理 query，避免刷新重复弹窗。
2. 管理员处理流程
   1. 管理员查看举报中心时，每条举报按照该举报的 `forum_id` 单独判断权限。
   2. 当前管理员拥有该 `forum_id` 权限时，该条举报显示管理员专用信息和处理入口；没有权限时，该条举报表现与一般用户一致。
   3. 管理员可以筛选“只显示我的待处理”，仅展示自己有权限板块内未处理的举报。
   4. 管理员确认内容后，可以选择处理方式。除“提示”外，处理后该举报项被标记为已处理。
      - 提示：调用 `userCheckPoster` / 后端对应检查接口，仅展示提示信息，不改变举报处理状态。
      - 忽略：直接将举报信息转为已处理状态，不执行帖子或用户操作。
      - 删帖：调用 `deletePostPoster`。
      - 删全：调用 `deleteAllPostPoster`，删除该楼层用户在该主题的所有回复。
      - 碎饼：调用 `userBanPoster`，永久 ban 用户。
      - 封禁饼干：调用 `userLockPoster`，短时间封禁用户。
      - 忽略只需要二次确认，不需要管理员填写原因。
      - 删帖、删全、碎饼、封禁饼干需要弹出确认框，管理员必须填写处理原因，处理原因写入数据库。
      - 删帖和删全的确认框需要有“罚款olo”可选项，对应 `deletePostPoster` / `deleteAllPostPoster` 的 `reduce_olo` 参数。
   5. 管理员可以标注“？”选项，表示处理不确定，以引起其他管理员注意。该标记仅对拥有该 `forum_id` 权限的管理员可见、可改，且与已处理/未处理状态无关。
   6. 仅拥有该 `forum_id` 权限的管理员可见举报人近 7 天举报次数、被举报回复作者近 7 天被举报次数、处理人和处理原因。
3. 处理后反馈
   1. 当前项目无站内信等系统，用户只能自行浏览举报中心确认被举报情况。
4. 匿名约束
   1. 举报人和被举报人身份信息对所有角色均不可见。

## 5. 页面设计

### 5.1 举报清单页面
- 路由：`/accuse?thread_id=xxx&post_id=xxx&floor=xxx`
  - 带有 `thread_id`、`post_id` 和 `floor` query 参数时，页面加载后自动弹出新增举报 modal。
  - 不带参数时，正常展示举报清单。
- 页面布局：
  - 顶部左侧：举报中心标题、待处理数量。
  - 顶部右侧：开关文案从“只显示未处理”改为“只显示我的待处理”。关闭状态显示“全部举报”。
  - 内容区：每个举报项为 `AccuseItem.vue`，竖向排列。
  - 底部右侧：分页导航，每页 10 条。
- 排序规则：按数据库主键 `id` 倒序。
- 待处理数量口径：
  - 默认总待处理数可保留为全站待处理数。
  - 若前端需要与开关文案完全一致，接口可额外返回 `my_pending_count`，表示当前管理员有权限板块内的待处理数。

### 5.2 举报信息组件
- 组件：`resources/vue/Accuse/AccuseItem.vue`
- 显示内容：
  - 第一行左侧：处理状态（已处理、未处理）和创建时间，所有人可见。
  - 第一行右侧：管理员处理下拉菜单和“？”标记。仅当前用户拥有该条举报 `forum_id` 权限时显示。
  - 第二行：所属板块名称，前面显示“板块：”。后端只返回 `forum_id`，前端通过 `resources/stores/forums.ts` 的板块数据实时解析 `forum.name`；找不到时可显示 `板块：#forum_id` 作为降级。
  - 第三行：被举报回复的主题 title，前面显示“主题：”。
  - 第四行：被举报回复的 floor，前面显示“楼层：”。形如 `楼层：№101`。floor 是超链接，用新窗口跳转至相应主题页面；跳转 page 按每页 200 floor 计算，且 floor 从 0 开始，所以 page = `Math.floor(floor / 200) + 1`，hash 为 `#f_${floor}`。被举报回复作者近 7 天被举报次数放在楼层号后面，仅有权限管理员可见。
  - 第五行：举报理由，所有人可见。多个用户举报同一个 post 时展示多条理由；每条理由的举报日期后面显示该举报人近 7 天举报次数，仅有权限管理员可见。
  - 第六行：处理人和处理原因，仅有权限管理员可见。删帖时如果勾选“罚款olo”，也在此处展示。

### 5.3 新增举报 modal
- 组件：`resources/vue/Accuse/AccuseCreateModal.vue`
- 触发条件：路由带有 `thread_id`、`post_id` 和 `floor` query 参数时自动弹出。
- 用户不能在举报中心手动打开该 modal，因为 `post_id` 平时不可见，手动填写容易填错。
- 隐含字段：举报回复 ID `post_id`，不展示给用户。
- 展示字段：
  - 举报主题 ID，只展示，不允许编辑。
  - 举报回复 floor，只展示，不允许编辑。
  - 举报理由，`n-input` textarea，不少于 5 字。
- modal 底部：取消按钮、提交按钮。

## 6. 接口与数据要求
- `GET /api/accuses`
  - 入参：
    - `page`: 页码。
    - `pending_only`: 保留兼容，表示仅未处理。
    - `my_pending_only`: 新增建议参数，表示仅当前管理员有权限板块内未处理。也可以复用 `pending_only` 并变更语义，但需要同步修改前后端命名，避免旧语义混淆。
  - 返回每条举报必须包含 `forum_id`，用于前端显示板块名和按条判断权限。
  - 返回每条举报建议包含 `can_manage` 布尔值，由后端基于当前用户和该条 `forum_id` 计算。前端可以用 `can_manage` 作为 UI 权限来源，避免只依赖本地 user store。
- `POST /api/accuses`
  - 不接受可信 `forum_id` 入参。
  - 后端通过 `post_id` 找到回复，从回复写入 `accuses.forum_id`。
- `POST /api/accuses/{accuse}/hint`
  - 必须校验当前用户具备该举报 `forum_id` 的对应管理员权限。
- `POST /api/accuses/{accuse}/handle`
  - 必须校验当前用户具备该举报 `forum_id` 的对应管理员权限。
  - `ban` 操作仍按现有更高权限要求执行，但同样必须在 `admins.forums` 中包含该 `forum_id`。
- `PUT /api/accuses/{accuse}/uncertain`
  - 必须校验当前用户具备该举报 `forum_id` 权限。

## 7. 前端组件边界
- `AccusePage.vue`：举报清单页面容器，负责筛选、分页、query 参数触发弹窗、调用列表接口和处理接口。
- `AccuseItem.vue`：单条举报展示，负责处理状态、板块名称、主题、楼层链接、举报理由、管理员操作按钮、“？”标记和处理备注展示。
- `AccuseCreateModal.vue`：新增举报表单，负责展示不可编辑的 `thread_id` 和 `floor`、隐含保存 `post_id`、填写举报理由和前端校验。
- `AccuseHandleModal.vue`：管理员处理弹窗，负责删帖、删全、碎饼、封禁饼干时填写处理原因；删帖和删全时额外提供“罚款olo”选项。
- `resources/api/methods/accuse.ts`：维护举报接口类型，新增 `forum_id`、`can_manage` 和“我的待处理”筛选参数。
